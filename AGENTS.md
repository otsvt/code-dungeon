# Правила работы с проектом

- Отвечай на русском языке.
- Не изменяй файлы и не выполняй команды без прямой просьбы пользователя.
- Сначала изучай существующий код и структуру проекта.
- Не придумывай файлы, компоненты, типы и архитектуру, которых нет в репозитории.
- Давай один следующий шаг за раз.
- Перед кодом коротко объясняй, что делаем и зачем.
- Код показывай как предложение. Пользователь вносит изменения самостоятельно.
- Не переписывай существующую реализацию без явной необходимости.
- Phaser отвечает за игровую сцену и игровые объекты.
- React отвечает за HUD, GameOverlay и обычный интерфейс.
- Не создавай Phaser-объекты через JSX.
- Перед предложением следующего шага проверяй реальные файлы проекта.
- Не изменяй файлы и не выполняй команды без прямой просьбы пользователя.
- По умолчанию только анализируй код, объясняй и предлагай изменения.
- Код показывает как предложение, пользователь вносит его самостоятельно.

# Документация проекта

Основные документы:

- `docs/DEVELOPMENT_PLAN_GAMEPLAY_FIRST.md` — актуальный порядок разработки по пути игрока.
- `docs/MVP_SCOPE.md` — актуальный состав MVP и игровые правила.
- `docs/VISUAL_DIRECTION.md` — визуальное направление и разделение React / Phaser.
- `docs/PROJECT_CONCEPT.md` — общая концепция проекта и долгосрочные идеи.
- `docs/DEVELOPMENT_PLAN.md` — старый архитектурный план, используется только как дополнительный технический материал.

# Приоритет источников

Прямая инструкция пользователя в текущем чате имеет наивысший приоритет.

Для определения текущего состояния реализации источником истины является реальный код репозитория.

Для определения следующего этапа и запланированного поведения используй документы в таком порядке:

1. `docs/DEVELOPMENT_PLAN_GAMEPLAY_FIRST.md`
2. `docs/MVP_SCOPE.md`
3. `docs/VISUAL_DIRECTION.md`
4. `docs/PROJECT_CONCEPT.md`
5. `docs/DEVELOPMENT_PLAN.md`

`docs/DEVELOPMENT_PLAN_GAMEPLAY_FIRST.md` заменяет старый общий порядок разработки из `docs/DEVELOPMENT_PLAN.md`.

Не используй `docs/DEVELOPMENT_PLAN.md` для определения текущей последовательности этапов, если он противоречит gameplay-first плану.

# Работа с противоречиями

Если документы противоречат друг другу:

- не смешивай разные версии решений;
- используй документ с более высоким приоритетом;
- явно укажи, какое решение было выбрано;
- не восстанавливай отменённые механики из старого документа.

Если код отличается от документации:

- сначала опиши, что реально реализовано в коде;
- затем объясни, чего не хватает до запланированного результата;
- не утверждай, что этап выполнен, если в коде отсутствуют его обязательные части.

Если противоречие нельзя однозначно разрешить по приоритетам, остановись и задай пользователю один конкретный вопрос.

# Формат работы

Перед началом каждого нового этапа:

1. Назови номер и название текущего этапа.
2. Составь полный план этапа в виде checklist с пунктами `- [ ]`.
3. Включай в checklist содержательные и проверяемые результаты, а не отдельные импорты, запятые и другие мелкие синтаксические действия.
4. Сверь checklist с реальным кодом и актуальной документацией.
5. Проходи checklist последовательно, предлагая по одному содержательному шагу за раз.
6. В конце этапа снова покажи checklist для финальной проверки; пользователь сам отмечает выполненные пункты.
7. Не переходи к следующему этапу, пока пользователь не подтвердил финальный checklist текущего этапа.

Перед каждым следующим шагом:

1. Проверь связанные файлы проекта.
2. Сверься с актуальным этапом в `DEVELOPMENT_PLAN_GAMEPLAY_FIRST.md`.
3. Коротко перечисли, что уже сделано.
4. Предложи только один следующий шаг.
5. Не переходи к следующему шагу, пока пользователь не подтвердил выполнение текущего.
