# Code Dungeon — Gameplay-First Development Plan

> **Статус документа:** рабочий и изменяемый. Он фиксирует актуальное направление разработки, но не является финальным набором неизменных условий. По ходу работы этапы, механики и требования могут добавляться, изменяться или удаляться по решению владельца проекта. После такого решения связанные документы должны быть синхронизированы с новой версией.

## 1. Назначение документа

Этот документ заменяет старый абстрактный development plan для текущей разработки MVP.

Здесь план идёт не от архитектурных слоёв, папок, Zustand, storage или backend-ready рассуждений, а от **пути игрока** и **игровой логики**.

Главный принцип:

```txt
игрок открыл игру
→ увидел экран
→ нажал кнопку
→ попал в следующий игровой шаг
→ совершил действие
→ получил игровой результат
→ пошёл дальше
```

Технические инструменты вроде Zustand, localStorage, services, repository, adapters и прочего не являются самостоятельными этапами разработки. Они добавляются внутри конкретного игрового шага, когда без них становится неудобно.

## 2. Что уже считаем сделанным

### Уже сделано

- Next.js-проект создан.
- Phaser Risk Spike сделан.
- Проверено, что Phaser запускается внутри Next.js.
- Проверено, что React overlay можно показывать поверх Phaser.
- Проверены базовые переходы, двери, сцена, герой, overlay и простое взаимодействие.
- Визуальное направление Code Dungeon зафиксировано отдельно.

### Важное решение

**Spike не трогаем.**

Он был техническим тестом. Он не является production-основой игры.

Дальше production-flow строится заново:

```txt
/
→ /run-setup
→ /game
→ /result
```

---

## 3. Главная цель MVP

MVP считается готовым, если игрок может пройти полный run:

```txt
Главное меню
→ выбор языка
→ Новый run
→ Run Setup
→ выбор пула вопросов
→ старт run
→ стартовая комната
→ стартовый баф через алтарь
→ выбор знака над дверью
→ обычная комната
→ challenge overlay
→ ответы на вопросы
→ итог комнаты
→ баф / дебаф / ничего
→ выбор следующего знака
→ несколько комнат
→ HR-комната
→ финальное собеседование
→ результат run
→ новый run или меню
```

Главный запрет:

```txt
не расширять проект,
пока не работает полный core gameplay loop
```

### Формат прохождения каждого этапа

Каждый этап начинается с полного рабочего плана в виде checklist:

```md
- [ ] содержательный результат 1
- [ ] содержательный результат 2
- [ ] содержательный результат 3
```

Пункты checklist должны описывать проверяемые результаты для игрока, game state, Phaser или React UI. Отдельные импорты, переименования, запятые и другие мелкие синтаксические действия не считаются самостоятельными шагами плана.

Checklist составляется до начала реализации и может уточняться по решению владельца проекта. Работа идёт последовательно по одному содержательному пункту за раз.

В конце этапа тот же checklist используется для финальной проверки. Пользователь сам отмечает выполненные пункты. Переход к следующему этапу происходит только после подтверждения финального checklist.

# 5. Этап 1 — Локализация RU / EN

## С чего начинаем

В главном меню уже есть переключение RU / EN, поэтому язык нельзя откладывать на потом.

Локализация должна быть заложена с первого экрана.

## Что делаем

Создаём минимальную систему языка:

```txt
текущий язык: ru / en
переключение языка
словарь текстов
функция получения текста
```

Используем i18n-библиотеку.

## Что должно быть локализовано сразу

```txt
главное меню
Run Setup
кнопки
сложности
HUD-лейблы
названия бафов
названия дебафов
названия HR-эффектов
результаты комнаты
финальный результат
```

Названия технологий не переводим:

```txt
JavaScript
TypeScript
HTML
CSS
React
Vue
Git
```

## Результат этапа

Можно переключить RU / EN, и текст интерфейса меняется.

## Проверка

```txt
открыть /
нажать EN
"Новый run" стал "New run"
нажать RU
вернулось "Новый run"
```

# 6. Этап 2 — Главное меню

## С чего начинаем

Первый экран, который видит игрок.

## Что делаем

Страница `/`.

На экране:

```txt
CODE DUNGEON

Новый run
Продолжить
RU / EN
```

Все тексты берутся из словаря локализации.

## Действия игрока

```txt
нажал RU / EN → язык изменился
нажал Новый run → переход в Run Setup
нажал Продолжить → если есть незавершённый run, переход в /game
```

На этом этапе кнопка `Продолжить` может быть disabled, если сохранения ещё нет.

## Что не делаем

```txt
логин
профиль
leaderboard
магазин
настройки аккаунта
достижения
```

## Результат этапа

Игрок открывает игру, видит понятное меню, может переключить язык и начать новый run.

## Проверка

```txt
/
→ видим меню
→ переключаем язык
→ нажимаем Новый run
→ попадаем в /run-setup
```

# 7. Этап 3 — Run Setup

## С чего начинаем

После нажатия `Новый run` игрок должен попасть на экран подготовки забега.

## Что делаем

Страница `/run-setup`.

На экране:

```txt
Формат пула:
- Frontend
- Custom

Технологии:
- JavaScript
- TypeScript
- HTML
- CSS
- React
- Vue
- Git

Кнопка:
- Начать run
```

Все тексты интерфейса берутся из словаря локализации.

## Действия игрока

```txt
выбрать Frontend или Custom
выбрать технологии, если Custom
нажать Начать run
```

## Что происходит при нажатии `Начать run`

Создаётся стартовая настройка run:

```txt
poolMode
technologies
```

Язык интерфейса остаётся глобальной настройкой приложения и не входит в настройки run.

Выбор базовой сложности в Run Setup пока не используется. Сложность конкретных вопросов и влияние игровых эффектов на следующие комнаты остаются частью game logic.

После этого игрок переходит на `/game`.

## Что не делаем

```txt
сохранение пресетов
backend
аккаунт
несколько готовых направлений
выбор базовой сложности
```

## Результат этапа

Игрок может настроить run и перейти в игру.

## Проверка

```txt
/
→ Новый run
→ /run-setup
→ выбрать Frontend
→ Начать run
→ /game
```

# 8. Этап 4 — Переход в игру

## С чего начинаем

После Run Setup игрок должен попасть не в пустую страницу, а в игровой экран.

## Что делаем

При нажатии `Начать run`:

```txt
создаём начальное состояние run
переходим на /game
передаём выбранные настройки в игру
```

На этом этапе можно хранить состояние временно в памяти приложения. Сохранение в localStorage не является целью этапа.

## Результат этапа

Путь `Меню → Run Setup → Game` работает.

## Проверка

```txt
выбрать параметры
→ нажать Начать run
→ открылась /game
→ настройки run доступны игровому экрану
```

# 10. Этап 6 — HUD

## С чего начинаем

Игрок должен видеть состояние run.

## Что делаем

На `/game` показываем HUD:

```txt
Жизнь: 1/1
Бафы
Дебафы
Впечатление
Пауза / Меню
```

Все лейблы через локализацию.

Базовое поведение HUD:

```txt
иконка головы видна постоянно
вне испытания HUD открывается и закрывается по нажатию на иконку
во время активного испытания HUD принудительно открыт
во время активного испытания закрыть HUD нельзя
после завершения испытания ручное переключение снова доступно
```

## Что не показываем в HUD

```txt
текущий вопрос
варианты ответа
номер вопроса внутри комнаты
код
таймер
explanation
```

Эти данные относятся к Challenge Overlay.

## Результат этапа

У игры появляется постоянная кнопка HUD и раскрываемая панель состояния.

## Проверка

```txt
/game
→ видим иконку головы
→ нажимаем на иконку
→ HUD открылся
→ видим Жизнь 1/1
→ видим пустые бафы
→ видим пустые дебафы
→ видим Впечатление: ровное
```

# 12. Этап 8 — Стартовый баф через алтарь

## План этапа

- [ ] Зафиксировать список идентификаторов стартовых даров и состояние, запрещающее повторную выдачу.
- [ ] Добавить в game store действие, которое один раз выбирает и записывает случайный стартовый дар.
- [ ] Сделать алтарь интерактивным Phaser-объектом.
- [ ] Связать активацию алтаря в Phaser с изменением Zustand-состояния через событие.
- [ ] Добавить простую визуальную реакцию алтаря на активацию.
- [ ] Показать локализованное название полученного дара в HUD.
- [ ] Проверить, что повторная активация не выдаёт второй стартовый дар.
- [ ] Зафиксировать завершение стартового события через `startBuffGranted`, чтобы этап 9 мог показать знаки дверей.

## С чего начинаем

Игрок должен активировать алтарь и получить первый случайный баф.

## Что делаем

Игрок нажимает / активирует алтарь.

После активации:

```txt
выбирается случайный стартовый баф
баф появляется в HUD
алтарь показывает простую реакцию
в game state фиксируется завершение стартового события
```

Сами знаки над дверями появляются на этапе 9. Этап 8 только подготавливает для них состояние `startBuffGranted`.

## Стартовые бафы MVP

```txt
Remove Wrong Option
Second Chance
Door Insight
Easier Next Room
Skip Question
```

## Результат этапа

Run начинается с игрового события, а не просто с пустой комнаты.

## Проверка

```txt
стартовая комната
→ активировать алтарь
→ в HUD появился баф
→ повторная активация не выдаёт второй баф
→ startBuffGranted = true
```

# 13. Этап 9 — Знаки над дверями

## С чего начинаем

После стартового бафа и `startBuffGranted = true` игрок должен выбрать путь.

## Что делаем

Над доступными дверями появляются два знака:

```txt
?   ?
```

Если есть Door Insight, один или несколько знаков могут быть раскрыты:

```txt
Battle   ?
HR       ?
```

## Базовое поведение

```txt
до результата комнаты:
- двери видны
- знаков нет
- дверь неактивна

после результата комнаты:
- появляются знаки
- игрок выбирает знак
- знак вспыхивает
- начинается переход
```

## Что не делаем

```txt
анимацию открытия дверей
ключи
сложную карту
3–4 двери
секретные двери
```

## Результат этапа

Появляется первый roguelike-выбор пути.

## Проверка

```txt
получили стартовый баф
→ появились 2 знака
→ нажали знак
→ игра готова перейти в следующую комнату
```

# 14. Этап 10 — Переход между комнатами

## С чего начинаем

После выбора знака игрок должен попасть в следующую комнату.

Flow перехода:

```txt
выбор знака
→ знак вспыхивает
→ fade out
→ выбирается следующая комната
```

## Результат этапа

Игрок может выйти из стартовой комнаты и попасть в следующую.

## Проверка

```txt
стартовая комната
→ стартовый баф
→ выбор знака
→ fade
→ Battle Room
```

# 17. Этап 13 — Challenge Overlay

## С чего начинаем

Теперь Battle Room должна открывать вопрос вместо dev-кнопок.

## Что делаем

При активации алтаря:

```txt
алтарь активируется
→ появляется символ технологии / портрет духа
→ открывается Challenge Overlay
→ показывается вопрос
```

Overlay должен поддерживать:

```txt
длинный вопрос
code block
4–6 вариантов ответа
длинные варианты
номер вопроса внутри комнаты
технологию
тип вопроса
сложность
портрет духа
```

## Главное правило

После отдельного ответа не показываем:

```txt
правильно / неправильно
correct answer
explanation
```

Ответ записывается, затем открывается следующий вопрос.

## Результат этапа

Игрок впервые проходит programming-испытание.

## Проверка

```txt
Battle Room
→ активировать алтарь
→ увидеть вопрос
→ выбрать ответ
→ ответ записался
→ correct/wrong не показали
```

# 18. Этап 14 — Итог Battle Room

## С чего начинаем

Теперь комната должна считать результат по ответам игрока.

## Что делаем

Комната содержит 1–3 вопроса.

После последнего ответа:

```txt
challenge overlay закрывается
→ считается общий результат комнаты
→ алтарь выдаёт итог
→ баф / дебаф / ничего
→ меняется Впечатление
→ появляются знаки над дверями
```

## Логика результата

```txt
0 ошибок → идеально → баф → Впечатление выше
часть ошибок → обычно → ничего → Впечатление без изменений
провал → дебаф → Впечатление ниже
```

## Результат этапа

Battle Room становится полноценной игровой комнатой.

## Проверка

```txt
ответить на все вопросы комнаты
→ получить общий итог
→ увидеть эффект в HUD
→ увидеть изменение Впечатления
→ увидеть знаки дверей
```

# 20. Этап 16 — HR Room

## С чего начинаем

Добавляем первый небоевой тип комнаты.

## Что делаем

HR Room отличается от Battle Room:

```txt
в центре HR-стол
нет programming-вопроса
открывается HR overlay
HR смотрит на Впечатление
```

## Логика HR

```txt
Слабое впечатление → HR-дебаф
Ровное впечатление → ничего
Сильное впечатление → HR-баф
```

## MVP HR-эффекты

```txt
HR-бафы:
- Хороший контакт
- Уверенная подача

HR-дебафы:
- Красный флаг
- Неловкая пауза
```

## Результат этапа

Run уже не состоит только из вопросов. Появляется отдельное HR-событие.

## Проверка

```txt
попали в HR Room
→ активировали HR-стол
→ получили HR-баф / HR-дебаф / ничего
→ появились знаки дверей
```

# 21. Этап 17 — Рабочие бафы и дебафы

## С чего начинаем

До этого эффекты могли просто висеть в HUD. Теперь они должны реально менять gameplay.

## Что делаем

Минимально подключаем эффекты:

```txt
Remove Wrong Option → убирает один неправильный вариант
Second Chance → прощает одну ошибку
Door Insight → раскрывает знак двери
Easier Next Room → снижает сложность следующей комнаты
Harder Next Room → повышает сложность следующей комнаты
```

## HR-эффекты

```txt
Хороший контакт → блокирует следующий HR-дебаф
Уверенная подача → следующий провал комнаты не ухудшает Впечатление
Красный флаг → следующее идеальное прохождение не улучшает Впечатление
Неловкая пауза → следующий провал сразу ставит Слабое впечатление
```

## Результат этапа

Игрок чувствует, что собирает билд, а не просто отвечает на тесты.

## Проверка

```txt
Door Insight → один знак двери раскрыт
Remove Wrong Option → вариантов ответа стало меньше
Harder Next Room → следующий вопрос сложнее
```

# 23. Этап 19 — Финальное собеседование

## С чего начинаем

Финальная комната должна не просто отображаться, а запускать серию вопросов.

## Что делаем

Финал использует те же типы вопросов, что и обычные комнаты, но вопросы сложнее и формулировки душнее.

## Влияние Впечатления

```txt
Сильное Впечатление → 5 вопросов / можно 2 ошибки
Ровное Впечатление → 6 вопросов / можно 1 ошибку
Слабое Впечатление → 7 вопросов / можно 1 ошибку
```

## Возможные исходы

```txt
Passed
Failed
```

## Результат этапа

Игрок может выиграть или проиграть run.

## Проверка

```txt
финальная комната
→ серия вопросов
→ подсчёт результата
→ passed или failed
```

## 32. Что нельзя делать до полного playable run

До этапа полного playable run нельзя уходить в:

```txt

админку

sandbox запуска кода

```

Можно готовить код так, чтобы эти вещи было легче добавить позже, но нельзя делать их вместо core gameplay loop.

## 33. Главная формула разработки

Каждый этап должен отвечать на три вопроса:

```txt
1. Что игрок теперь может сделать?
2. Какой игровой результат он получает?
3. Как мы проверяем, что это работает?
```

Если этап не даёт нового игрового действия или нового игрового результата, значит это не отдельный этап разработки, а внутренняя техническая задача.
