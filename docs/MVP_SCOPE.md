# MVP Scope

> **Статус документа:** рабочий и изменяемый. Он описывает актуальный состав MVP, но не является финальным набором неизменных условий. По ходу разработки функции, правила и границы MVP могут добавляться, изменяться или удаляться по решению владельца проекта. После такого решения связанные документы должны быть синхронизированы с новой версией.

## 1. Назначение документа

Этот документ фиксирует объём первой MVP-версии проекта.

MVP в рамках этого проекта — это не финальный продукт и не коммерческий релиз. MVP — это первый полноценный играбельный вертикальный срез, который доказывает, что основная идея работает:

> Игрок начинает run, проходит случайные комнаты, отвечает на programming-испытания, получает бафы/дебафы, выбирает знаки над дверями, доходит до финального собеседования и получает результат забега.

Главная задача MVP — проверить core gameplay loop, а не реализовать все будущие системы.

---

## 2. Главный принцип MVP

MVP должен быть маленьким, но настоящим.

Нельзя делать просто набор экранов, макет или красивое меню без игрового цикла. В MVP обязательно должен быть один законченный run от старта до финала.

Правильный MVP:

- можно начать run;
- можно выбрать стартовые настройки;
- можно пройти несколько комнат;
- можно получить вопросы;
- можно ошибиться;
- можно получить баф или дебаф;
- можно выбрать знак над дверью;
- можно дойти до финального собеседования;
- можно победить или проиграть;
- можно увидеть результат run.

Неправильный MVP:

- есть только меню;
- есть только Phaser-сцена без игровой логики;
- есть только база вопросов;
- есть только UI без реального run;
- есть только технический прототип движения персонажа.

---

## 3. Цель MVP

Цель первой версии — собрать playable prototype, который проверяет три главные гипотезы:

### 1. Gameplay-гипотеза

Отвечать на programming-вопросы внутри roguelike-run может быть интересно как игра, а не только как тест.

### 2. Системная гипотеза

Комнаты, знаки над дверями, бафы, дебафы, статус **Впечатление**, HR-события и финальное собеседование могут работать как единый игровой цикл.

### 3. Техническая гипотеза

Next.js и Phaser можно нормально связать так, чтобы продуктовая оболочка, React overlay и игровая сцена не мешали друг другу.

---

## 4. Что считается готовым MVP

MVP считается готовым, если игрок может пройти полный run:

1. Открыть главное меню.
2. Выбрать язык интерфейса.
3. Перейти в Run Setup.
4. Выбрать способ формирования пула вопросов.
5. Выбрать готовое направление или кастомный пул.
6. Начать run.
7. Попасть в первую стартовую комнату.
8. Подойти к алтарю и получить случайный стартовый баф.
9. Увидеть баф в HUD.
10. Выбрать знак над дверью.
11. Попасть в обычную комнату.
12. Активировать центральный объект.
13. Пройти programming-испытание через React overlay.
14. Получить итог комнаты: баф, дебаф или ничего.
15. Увидеть появление знаков над дверями.
16. Несколько раз повторить цикл комнат.
17. Попасть в HR-комнату и получить HR-исход через Впечатление.
18. Дойти до финального собеседования.
19. Пройти финальное испытание.
20. Увидеть итог run.
21. Вернуться в меню или начать новый run.

---

## 5. Обязательный игровой цикл MVP

В MVP должен быть реализован следующий минимальный цикл:

```txt
Main Menu
→ Run Setup
→ Start Room
→ Random Start Buff
→ Door Sign Choice
→ Room
→ Challenge Overlay
→ Room Outcome
→ Buff / Debuff / Nothing
→ Door Sign Choice
→ Next Room
→ HR Room / Battle Room
→ Final Interview
→ Run Result
```

Этот цикл важнее любых дополнительных фич.

Если какая-то фича не помогает этому циклу, она не входит в MVP.

Важно: стартовый баф не выдаётся отдельным pop-up до сцены. Игрок получает его внутри первой комнаты через алтарь.

---

## 6. Главное меню

В MVP должен быть простой главный экран.

На нём доступны:

- начать новый run;
- продолжить незавершённый run, если он есть;
- выбрать язык интерфейса RU / EN.

Кнопка “Новый run” ведёт в Run Setup.

Необязательно для MVP:

- авторизация;
- профиль;
- leaderboard;
- магазин;
- достижения;
- настройки аккаунта;
- сложная главная страница;
- промо-лендинг.

---

## 7. Настройка run

Перед началом run игрок выбирает:

- способ формирования пула вопросов.

Язык интерфейса выбирается глобально в главном меню и не входит в настройки run.

Выбора базовой сложности в Run Setup пока нет.

### Способ формирования пула

В MVP должны быть два способа:

### Готовое направление

Игрок выбирает направление. Для MVP достаточно одного готового направления:

- Frontend.

Frontend-направление включает стартовый пул:

- JavaScript;
- TypeScript;
- HTML;
- CSS;
- React;
- Vue;
- Git.

### Кастомный пул

Игрок может вручную выбрать технологии из доступного списка.

Для MVP кастомный пул может использовать тот же набор технологий:

- JavaScript;
- TypeScript;
- HTML;
- CSS;
- React;
- Vue;
- Git.

Необязательно для MVP:

- Backend-направление;
- Fullstack-направление;
- DevOps-направление;
- Algorithms-направление;
- большое количество технологий;
- сохранение пользовательских пресетов.

---

## 8. Сложность вопросов

Игрок пока не выбирает базовую сложность перед началом run.

Сложность определяется игровыми правилами и данными вопросов. Внутренняя система может учитывать:

- уровень вопросов;
- количество вопросов в комнате;
- шанс выпадения сложных комнат;
- активные дебафы.

Для MVP не нужен отдельный параметр сложности в `RunSettings`. Достаточно рабочей внутренней системы, которую позже можно дополнить пользовательским выбором Easy / Normal / Hard.

---

## 9. Игровая сцена

MVP должен использовать Phaser для игровой сцены.

В игровой сцене должны быть:

- комната;
- персонаж;
- центральный объект или место появления духа;
- двери;
- базовая камера;
- простое перемещение или переходы;
- визуальное появление события/духа;
- переход между комнатами.

Для MVP не требуется:

- сложная боевая анимация;
- полноценная изометрическая графика уровня коммерческой игры;
- большое количество скинов;
- сложные эффекты;
- физика;
- pathfinding;
- полноценная 3D-сцена.

Важно: сцена должна выглядеть как ранний игровой прототип, а не как пустой технический canvas.

---

## 10. Комнаты MVP

В MVP нужны не все типы комнат из концепта, а минимальный набор для проверки gameplay loop.

Обязательные типы комнат:

### 1. Start Buff Room

Стартовая комната run.

В центре стоит алтарь. Игрок подходит к нему и получает случайный стартовый баф из Start Pool.

Стартовая комната не содержит испытания. После получения бафа появляются знаки над дверями, и игрок выбирает следующую комнату.

### 2. Battle Room

Обычная комната с духом языка или технологии.

Игрок активирует центральный объект, проходит programming-испытание через React overlay, затем получает итог комнаты:

```txt
идеальное прохождение → баф
обычное прохождение → ничего
провал → дебаф
```

### 3. HR Room

Отдельная событийная комната с HR-персонажем.

В центре вместо алтаря стоит HR-стол. HR проверяет статус **Впечатление** и выдаёт HR-исход:

```txt
Слабое впечатление → HR-дебаф
Ровное впечатление → ничего
Сильное впечатление → HR-баф
```

### 4. Final Interview Room

Финальная комната run.

В центре стоит финальный стол с двумя интервьюерами: HR и техническим интервьюером. Игрок проходит финальное собеседование, после чего открывается экран результата run.

Не входит в обязательные комнаты MVP:

- Reward Room как отдельная комната;
- Mood Room как отдельная комната;
- секретные комнаты;
- бывший друг;
- наставник;
- токсичный коллега;
- полноценные мини-боссы;
- разные финальные боссы;
- разные компании;
- сложная карта мира.

Reward Room и Mood Room можно сохранить как идеи для future expansion, но текущий MVP-цикл строится без них.

---

## 11. Структура run

Для MVP run должен быть коротким.

Предлагаемый размер run:

- 1 стартовая комната без испытания;
- 5 основных комнат;
- 1 финальная комната.

Итого:

```txt
start room + 5 rooms + final interview
```

Это достаточно мало, чтобы быстро тестировать игру, но достаточно много, чтобы проверить цикл:

- стартовый баф;
- несколько событий;
- накопление эффектов;
- изменение Впечатления;
- финал.

Позже run можно расширить до нескольких актов, веток, мини-боссов и большой карты.

---

## 12. Выбор дверей и знаки пути

После завершения комнаты игрок выбирает следующий путь через знаки над дверями.

Для MVP достаточно 2 доступных дверей после каждой комнаты.

Важно: двери визуально не открываются. Основной интерактивный элемент — знак над дверью.

Базовое поведение:

```txt
до результата комнаты:
- дверь видна;
- знака нет;
- дверь неактивна;

после результата комнаты:
- над доступными дверями появляются знаки;
- знак может быть “?” или раскрытым типом комнаты;
- игрок выбирает знак / дверь;
- знак вспыхивает;
- начинается fade-переход.
```

Знаки могут вести в разные типы комнат:

- Battle Room;
- HR Room;
- Final Interview Room, если run дошёл до финала.

Можно добавить простой баф `Door Insight`, который показывает тип одной или нескольких дверей.

Необязательно для MVP:

- 3–4 двери;
- сложная карта как в Slay the Spire;
- визуальная схема всего пути;
- ключи;
- запертые двери;
- секретные двери;
- анимация открытия дверей.

---

## 13. Испытания MVP

В MVP используются задания без livecoding.

Обязательные типы испытаний:

### Quiz

Обычный вопрос с вариантами ответа.

### True / False

Утверждение, которое нужно подтвердить или опровергнуть.

### Code Output

Вопрос “что выведет код?”.

### Find the Bug

Нужно найти ошибку в коде или описании.

Дополнительные типы, если успеваем:

### Choose Correct Fragment

Выбор правильного фрагмента кода.

### Order Steps

Расстановка шагов в правильном порядке.

Если сроки или сложность начнут раздуваться, MVP можно сузить до четырёх обязательных типов:

- Quiz;
- True / False;
- Code Output;
- Find the Bug.

---

## 14. Контент вопросов MVP

Для MVP нужен небольшой, но достаточный набор вопросов.

Минимально допустимый объём:

- 40 вопросов.

Желательный объём:

- 60–80 вопросов.

Стартовое распределение:

- JavaScript — 15;
- TypeScript — 10;
- HTML — 5;
- CSS — 10;
- React — 10;
- Vue — 10;
- Git — 5.

Вопросы должны быть структурированы в коде, а не захардкожены в компонентах.

Каждый вопрос должен иметь:

- id;
- type;
- technology;
- difficulty;
- tags;
- prompt;
- code, если нужен;
- options;
- correct answer;
- explanation;
- locale.

Для MVP вопросы можно хранить в проекте как TypeScript/JSON-данные.

База данных для вопросов не обязательна в первой версии.

---

## 15. Духи языков и технологий

В MVP нужны простые духи технологий.

Минимальный набор:

- JavaScript Spirit;
- TypeScript Spirit;
- CSS Spirit;
- React Spirit;
- Vue Spirit;
- Git Spirit.

Дух в MVP — это не сложный NPC и не полноценная модель в комнате.

Минимально достаточно:

- имя;
- технология;
- портрет или силуэт для overlay;
- короткая реплика перед испытанием;
- связь с вопросом / типом испытания.

После каждого отдельного ответа дух не обязан показывать реакцию “правильно/неправильно”. Ответ записывается, затем открывается следующий вопрос. Итог считается после завершения всей комнаты.

Необязательно для MVP:

- длинные диалоги;
- сюжетные ветки;
- озвучка;
- уникальные анимации для каждого духа;
- сложные характеры персонажей.

---

## 16. Бафы MVP

В MVP нужны несколько простых бафов, которые реально меняют правила игры.

Минимальный набор бафов:

### Remove Wrong Option

Убирает один неправильный вариант ответа.

### Second Chance

Даёт право на одну ошибку без провала комнаты.

### Door Insight

Показывает тип одной или нескольких дверей.

### Easier Next Room

Снижает сложность следующей комнаты.

### Skip Question

Позволяет один раз заменить вопрос.

Для MVP достаточно 5 бафов.

Необязательно для MVP:

- десятки бафов;
- сложные синергии;
- редкости;
- прокачка бафов;
- визуальный инвентарь с большим количеством предметов.

---

## 17. Дебафы MVP

В MVP нужны несколько негативных эффектов.

Минимальный набор дебафов:

### Timer Pressure

Добавляет таймер на следующую комнату.

### Hidden Door Info

Скрывает информацию о дверях.

### Harder Next Room

Повышает сложность следующей комнаты.

### Extra Technology

Добавляет случайную технологию в пул на несколько комнат.

### Fewer Hints

Отключает помощь или подсказку на следующую комнату.

Для MVP достаточно 5 дебафов.

Необязательно для MVP:

- десятки дебафов;
- сложные дебаф-пулы;
- уникальные дебафы для каждого персонажа;
- многоуровневые дебафы.

---

## 18. Супер-проклятия / Last Chance

Супер-проклятия и Last Chance остаются идеей для расширения, но не являются обязательной частью текущего MVP.

В HUD можно оставить место под будущий статус последнего шанса, но core loop MVP не должен зависеть от этой механики.

Для текущего MVP достаточно:

- обычных бафов;
- обычных дебафов;
- HR-бафов;
- HR-дебафов;
- статуса Впечатление.

Опционально, если core loop уже работает и остаётся время, можно добавить один простой Last Chance-эффект:

### Final Warning

Игрок получает предупреждение после критического провала.

Эффект:

- следующая критическая ошибка завершает run;
- финальное собеседование становится сложнее;
- часть помощи отключается до конца run.

Но это не blocker для MVP.

---

## 19. Впечатление MVP

В текущем MVP вместо отдельной Mood Room используется один soft-skill статус — **Впечатление**.

Это не обычный баф и не дебаф. Это отдельный статус, который влияет на HR и финальное собеседование.

Уровни:

```txt
-1 → Слабое впечатление
 0 → Ровное впечатление
+1 → Сильное впечатление
```

Как меняется:

```txt
провал комнаты → Впечатление ухудшается
обычное прохождение → Впечатление не меняется
идеальное прохождение → Впечатление улучшается
```

Как влияет на HR:

```txt
Слабое впечатление → HR-дебаф
Ровное впечатление → ничего
Сильное впечатление → HR-баф
```

Как влияет на финал:

```txt
Сильное впечатление → финал мягче
Ровное впечатление → стандартный финал
Слабое впечатление → финал жёстче
```

Good Mood, Anxiety, Focus и другие состояния можно сохранить как future expansion, но они не входят в обязательный MVP.

---

## 20. HR-комната MVP

HR-комната — отдельный тип события.

В центре комнаты стоит HR-стол. Игрок подходит к столу, после чего открывается HR overlay.

В MVP HR не даёт вопрос с выбором ответа. HR проверяет текущее **Впечатление** и выдаёт исход.

Минимальная логика:

```txt
Слабое впечатление:
  HR даёт HR-дебаф

Ровное впечатление:
  HR ничего не даёт

Сильное впечатление:
  HR даёт HR-баф
```

MVP-набор HR-эффектов:

### HR-бафы

- Хороший контакт;
- Уверенная подача.

### HR-дебафы

- Красный флаг;
- Неловкая пауза.

Черновая логика:

```txt
Хороший контакт → блокирует следующий HR-дебаф
Уверенная подача → следующий провал комнаты не ухудшает Впечатление
Красный флаг → следующее идеальное прохождение не улучшает Впечатление
Неловкая пауза → следующий провал сразу ставит Слабое впечатление
```

---

## 21. Финальное собеседование MVP

Финальное собеседование — обязательная часть MVP.

Минимальная версия:

- финальная комната;
- финальный стол;
- два интервьюера: HR и технический интервьюер;
- несколько вопросов подряд;
- вопросы могут быть из разных технологий выбранного пула;
- используются те же типы испытаний, что и в обычных комнатах;
- формулировки сложнее и “душнее”;
- накопленные эффекты и Впечатление влияют на финал;
- после прохождения показывается результат run.

Впечатление влияет на параметры финала:

```txt
Сильное впечатление:
- меньше вопросов;
- больше допустимых ошибок;
- меньше душных формулировок.

Ровное впечатление:
- стандартный финал.

Слабое впечатление:
- больше вопросов;
- меньше права на ошибку;
- больше вопросов с похожими вариантами ответа.
```

Пример MVP-настроек:

```txt
Сильное → 5 вопросов / можно 2 ошибки
Ровное → 6 вопросов / можно 1 ошибку
Слабое → 7 вопросов / можно 1 ошибку
```

Возможные результаты:

### Passed

Игрок успешно прошёл финальное собеседование.

Финальная фраза:

> “Спасибо, мы свяжемся с вами до конца недели.”

### Failed

Игрок провалил финальное собеседование.

Финальная фраза:

> “Мы приняли решение продолжить с другими кандидатами.”

Необязательно для MVP:

- несколько фаз финального босса;
- оффер;
- мета-прогрессия после финала;
- разные компании;
- сюжетные концовки.

---

## 22. Жизни и провал MVP

В MVP используется простая модель выживания.

Фиксируем:

```txt
жизнь: 1/1
```

Обычный провал комнаты не обязан сразу завершать run. Для MVP важнее, чтобы провалы накапливали последствия и усиливали драму забега.

Базовая логика:

- ошибка в отдельном вопросе записывается;
- результат считается после всей комнаты;
- провал комнаты даёт дебаф;
- идеальное прохождение даёт баф;
- обычное прохождение не даёт эффект;
- провал комнаты ухудшает Впечатление;
- финальное собеседование определяет passed / failed.

Для MVP можно считать комнату проваленной, если игрок ошибся в большинстве вопросов комнаты.

Пример:

- в комнате 1 вопрос — ошибка = провал комнаты;
- в комнате 3 вопроса — 2 ошибки = провал комнаты;
- идеальная комната — 0 ошибок.

Last Chance / супер-проклятие можно добавить позже, но оно не является обязательным условием MVP.

---

## 23. Результат комнаты

После каждого отдельного ответа не показываем:

- правильно / неправильно;
- correct answer;
- explanation.

Ответ записывается, затем открывается следующий вопрос.

Итог показывается только после завершения всей комнаты.

Минимальный flow:

```txt
последний вопрос завершён
→ challenge overlay закрывается
→ комната считает общий результат
→ алтарь активируется
→ появляется итог: баф / дебаф / ничего
→ если есть эффект, иконка улетает в HUD
→ появляются знаки над дверями
```

Результаты комнаты:

```txt
идеально → баф
обычно → ничего
провал → дебаф
```

`explanation` остаётся в структуре вопроса, но не показывается сразу после каждого ответа в MVP-flow. Позже его можно вывести в подробном result review или истории run.

---

## 24. Результат run

После завершения run игрок видит итог.

Минимально:

- победа или поражение;
- сколько комнат пройдено;
- сколько вопросов было;
- сколько правильных ответов;
- точность в процентах;
- какие технологии выпадали;
- какие бафы/дебафы были получены;
- финальная фраза.

Необязательно для MVP:

- глобальная статистика;
- аккаунт;
- сохранение на сервер;
- сравнение с другими игроками;
- replay;
- шаринг результата.

---

## 25. Локальное сохранение

В MVP можно использовать localStorage.

Сохранять минимально:

- последний результат run;
- простую историю последних run;
- глобальный язык интерфейса, отдельно от настроек run;
- последние настройки run.

Backend не обязателен для первой версии MVP.

Но архитектура не должна мешать будущему backend-сохранению.

---

## 26. Локализация MVP

В MVP нужно заложить архитектуру под два языка:

- русский;
- английский.

Необязательно сразу идеально перевести весь контент.

Минимально:

- интерфейс должен поддерживать ru/en;
- структура вопросов должна иметь locale;
- система должна позволять добавлять переводы.

Для MVP можно начать с русской версии контента и подготовленной архитектуры под английскую.

---

## 27. Product Layer MVP

В MVP product layer минимальный.

Обязательные страницы:

- `/` — стартовый экран;
- `/game` — страница с Phaser-сценой;
- `/result` или result overlay — результат run.

Необязательно:

- `/profile`;
- `/leaderboard`;
- `/admin`;
- `/login`;
- `/settings`;
- `/history`.

Эти страницы относятся к следующим этапам.

---

## 28. Game Layer MVP

В MVP game layer должен содержать:

- создание run;
- создание стартовой комнаты;
- выдачу случайного стартового бафа;
- генерацию комнат;
- генерацию знаков над дверями;
- выбор следующей комнаты;
- выбор вопроса;
- запись ответа игрока;
- подсчёт результата комнаты;
- применение бафов;
- применение дебафов;
- применение HR-эффектов;
- изменение Впечатления;
- завершение комнаты;
- запуск финального собеседования;
- подсчёт результата финала;
- завершение run.

Игровая логика не должна быть размазана по React-компонентам.

React отвечает за оболочку и UI-оверлеи.

Phaser отвечает за игровую сцену.

Game logic должна быть отдельным слоем, который можно тестировать отдельно.

## 29. UI MVP

В MVP нужны следующие UI-элементы:

- главное меню;
- экран настройки run;
- HUD во время run;
- challenge overlay;
- варианты ответа;
- отображение результата комнаты через алтарь / эффект;
- список активных бафов и дебафов;
- отображение Впечатления;
- знаки над дверями;
- HR overlay;
- final interview overlay;
- экран результата run.

UI должен быть простым, но аккуратным.

Необязательно:

- отдельное окно результата после каждого ответа;
- окно выбора награды;
- сложные анимации UI;
- drag-and-drop, если он не нужен типу вопроса;
- большой инвентарь;
- магазин;
- коллекция открытых вопросов;
- профиль игрока.

## 33. Критерии успеха MVP

MVP можно считать успешным, если:

- полный run можно пройти от начала до конца;
- игрок понимает, что происходит;
- вопросы реально влияют на прохождение;
- бафы и дебафы ощущаются в gameplay;
- финальное собеседование ощущается как цель run;
- после проигрыша или победы хочется попробовать ещё раз;
- архитектура не мешает расширению проекта;
- проект можно показать как серьёзный frontend/game/product prototype.

---

## 34. Главный запрет MVP

Главный запрет:

> Не расширять проект, пока не работает полный core gameplay loop.

Нельзя уходить в backend, админку, коллекции, авторизацию, сложный визуал, livecoding или мета-прогрессию, пока игрок не может пройти простой run от старта до финального собеседования.
