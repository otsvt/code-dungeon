# Development Plan

> **Статус документа:** рабочий и изменяемый. Он содержит дополнительный технический материал, но не является финальным набором неизменных условий. По ходу разработки этапы, архитектурные решения и требования могут добавляться, изменяться или удаляться по решению владельца проекта. После такого решения связанные документы должны быть синхронизированы с новой версией.

## 1. Назначение документа

Этот документ фиксирует порядок разработки MVP.

Цель плана — довести проект до первого playable prototype, где игрок может пройти run от стартового экрана до финального собеседования.

Документ опирается на уже зафиксированные решения:

- проект является игрой в первую очередь;
- жанр — browser programming roguelike;
- основной стек — Next.js, TypeScript, Phaser, Tailwind CSS;
- livecoding не входит в MVP;
- backend не входит в первый playable prototype;
- MVP должен содержать полный core gameplay loop;
- финальная цель run — собеседование на работу.

Главная задача разработки — не распылиться и не уйти в бесконечное проектирование. Сначала нужно доказать, что техническая и игровая основа работает.

---

## 2. Главный принцип разработки

Главный принцип:

> Сначала проверяем самые рискованные части, потом строим полноценную систему.

Самая неизвестная зона проекта — Phaser, игровая сцена, переходы между комнатами, связь Phaser с React/Next и отображение игровых событий.

Поэтому Phaser нужно проверить в самом начале, но не в отдельном проекте.

Phaser Risk Spike делается внутри основного Next.js-приложения, в отдельной sandbox/dev-зоне. Это позволит сразу тестировать Phaser в реальном окружении проекта: с тем же TypeScript, Tailwind, App Router, структурой папок и будущими ограничениями.

---

## 3. Backend-ready принцип

Backend не входит в первый playable prototype, но он точно должен появиться позже.

Архитектура MVP должна сразу предполагать backend, чтобы потом не пришлось переписывать весь проект.

Для этого:

- game domain не зависит от localStorage, IndexedDB, fetch, Prisma или backend;
- runtime state хранится отдельно от persistence;
- изменение run происходит через actions/reducer;
- сохранение идёт через repository/service слой;
- вопросы получаются через question service;
- результат run имеет сериализуемую структуру;
- localStorage используется только как временная реализация persistence layer;
- backend позже заменяет или дополняет persistence layer без переписывания UI, Phaser и game domain.

Главное правило:

> Компоненты, Phaser и game domain не должны знать, где физически лежат данные: в localStorage, IndexedDB, PostgreSQL или API.

Правильная зависимость:

```txt
UI / Phaser
↓
Zustand store
↓
game actions / reducer
↓
game domain
↓
repositories / services
↓
localStorage сейчас
↓
backend позже
```

## 4. Что такое Phaser Risk Spike

Phaser Risk Spike — это быстрый технический прототип внутри основного проекта.

Это не финальная игра, не MVP и не красивая сцена. Это проверка:

> Можем ли мы внутри основного Next.js-приложения запустить Phaser-сцену, показать комнату, персонажа, двери, духа, React-оверлей с вопросом и переход между комнатами?

Для spike можно использовать любые placeholders:

- квадраты;
- круги;
- простые цвета;
- временные PNG;
- текстовые заглушки;
- примитивные эффекты.

Красота не важна. Важно понять, что техническая связка работает.

Возможное расположение spike-зоны:

```txt
src/app/dev/phaser-spike/page.tsx
src/game/phaser-spike/
```

или:

```txt
src/app/_sandbox/phaser/page.tsx
src/game/sandbox/
```

После проверки spike-зону можно:

- удалить;
- оставить как playground для визуальных тестов;
- частично переиспользовать для production `/game`.

## 10. Этап 5 — State Management Architecture

Цель этапа: заранее определить, где хранится состояние run и кто имеет право его менять.

В проекте нужно разделить несколько видов состояния.

### Runtime state

Это состояние текущего run прямо сейчас:

- текущая комната;
- текущий вопрос;
- выбранный пул технологий;
- активные бафы;
- активные дебафы;
- Впечатление;
- активные HR-эффекты;
- текущая сложность;
- доступные двери;
- накопленные ответы текущей комнаты;
- результат текущей комнаты;
- статус финального собеседования.

Runtime state хранится в памяти приложения.

Для него используется:

```txt
Zustand store + game reducer/actions
```

### Game domain

Game domain — это чистая игровая логика.

Она не зависит от React, Phaser, browser API, localStorage, IndexedDB или backend.

Game domain должен работать как набор TypeScript-типов и чистых функций.

### Phaser state

Phaser не является источником истины.

Phaser отвечает за визуальное отображение:

- комнаты;
- персонажа;
- дверей;
- духа;
- переходов;
- простых визуальных эффектов.

Phaser может отправлять события в приложение, но не должен сам решать правила run.

### UI state

UI state — это состояние интерфейса:

- открыт ли overlay;
- какой modal показывается;
- выбранный вариант ответа;
- состояние кнопок;
- локальная форма настройки run.

Простые UI-состояния можно держать в `useState`.

### Persistent state

Persistent state — это данные, которые должны переживать перезагрузку:

- глобальный язык интерфейса, отдельно от настроек run;
- последние настройки run;
- последний результат run;
- короткая история run.

Для MVP используется localStorage через отдельный persistence layer.

Позже этот слой можно заменить на IndexedDB или backend.

## 11. Рекомендуемая модель состояния

Рекомендуемая модель:

```txt
React UI
  ↓ dispatch(action)
Zustand Store
  ↓ uses
Game Domain Reducer / Pure Functions
  ↓ returns new state
Zustand Store updates state
  ↓
React UI and Phaser receive new state
```

Пример направления событий:

```txt
Игрок нажал на дверь в Phaser
→ Phaser отправил событие DOOR_SELECTED
→ React/Zustand вызвал action chooseDoor
→ game domain сгенерировал следующую комнату
→ Zustand обновил gameState
→ Phaser получил новое состояние и показал новую комнату
```

Важно:

- React не должен вручную мутировать глубокое состояние run;
- Phaser не должен напрямую менять gameState;
- localStorage не должен вызываться из случайных компонентов;
- backend в будущем не должен ломать game domain.

- описать типы эффектов;
- описать типы источников эффектов;
- описать тип финального собеседования;
- описать сериализуемый тип результата run;
- создать actions для основных событий run;
- создать reducer или набор чистых функций;
- создать функцию создания run;
- создать функцию генерации комнаты;
- создать функцию выбора следующей комнаты;
- создать функцию выбора вопроса;
- создать функцию обработки ответа;
- создать функцию применения эффекта;
- создать функцию завершения комнаты;
- создать функцию завершения run.

Важно: названия конкретных эффектов пока не фиксируются.

На этом этапе используются описательные placeholder-эффекты:

- баф, который убирает один неправильный вариант ответа;
- баф, который даёт право на ошибку;
- баф, который показывает информацию о двери;
- баф, который снижает сложность следующей комнаты;

- дебаф, который добавляет таймер;

- дебаф, который повышает сложность следующей комнаты;
- дебаф, который добавляет случайную технологию в пул;
- дебаф, который отключает помощь;
- HR-эффект, который влияет на Впечатление;
- optional Last Chance-эффект, если core loop уже работает.

Результат этапа:

- можно создать run в коде;
- можно сгенерировать комнаты;
- можно выбрать вопрос;
- можно обработать ответ;
- можно применить эффект;
- можно дойти до финала как логическая система;
- game domain можно тестировать отдельно от React и Phaser;
- результат run можно позже отправить на backend.

## 13. Этап 7 — Repository / Service Layer

Цель этапа: подготовить проект к будущему backend без переписывания UI, Phaser и game domain.

В MVP repository/service слой может работать с локальными данными, но интерфейсы должны быть backend-ready.

Нужные сервисы:

```txt
runService
questionService
settingsService
progressService
```

### runService

Отвечает за сохранение и получение результатов run.

В MVP:

- сохраняет последний run локально;
- сохраняет короткую историю локально.

Позже:

- отправляет результат на backend;
- получает историю run пользователя;
- синхронизирует локальную историю с сервером.

### questionService

Отвечает за получение вопросов.

В MVP:

- берёт вопросы из локальных TypeScript/JSON-файлов.

Позже:

- получает вопросы из API;
- работает с PostgreSQL через backend;
- может использовать IndexedDB/cache;
- может учитывать модерацию и версии вопросов.

### settingsService

Отвечает за пользовательские настройки.

В MVP:

- хранит язык интерфейса;
- хранит последние настройки run.

Позже:

- синхронизирует настройки с аккаунтом пользователя.

### progressService

Отвечает за прогресс игрока.

В MVP:

- может отсутствовать или быть заглушкой.

Позже:

- хранит статистику;
- достижения;
- открытые вопросы;
- мета-прогрессию;
- пользовательский профиль.

Результат этапа:

- компоненты не знают, используется localStorage или backend;
- game domain не знает, откуда берутся вопросы;
- backend можно добавить через новые реализации сервисов;
- проект не привязан намертво к MVP-хранилищу.

## 16. Этап 10 — Playable Logic Loop

Цель этапа: собрать полный run как рабочий игровой цикл.

На этом этапе можно использовать простой React UI и placeholders, но логика должна быть настоящей.

Задачи:

- создать run;
- сгенерировать стартовую комнату;
- выдать стартовый баф через алтарь;
- показать текущую комнату;
- показать текущий вопрос;
- обработать ответ;
- записать ответы без немедленного correct/wrong;
- посчитать общий результат комнаты;
- применить баф, дебаф или ничего;
- показать активные эффекты;
- показать знаки над дверями;
- перейти в следующую комнату;
- пройти 5 комнат;
- перейти к финальному собеседованию;
- показать результат run.

Результат этапа:

- полный run можно пройти без финальной графики;
- core gameplay loop работает;
- можно тестировать баланс;
- можно понять, где логика ломается.

Важно: этот этап не заменяет Phaser. Phaser уже проверен spike-этапом. Здесь задача — собрать настоящую игровую систему.

## 17. Этап 11 — Phaser Production Integration

Цель этапа: заменить временное представление комнат нормальной Phaser-сценой.

Задачи:

- создать GameScene или RoomScene;
- создать слой комнаты;
- создать слой персонажа;
- создать слой дверей;

- добавить переходы между комнатами;
- добавить визуальное появление духа;
- связать Phaser-события с game domain;
- оставить вопросы и сложный UI в React overlay;
- не переносить всю бизнес-логику в Phaser.

Результат этапа:

- игрок видит игровую сцену;
- комната отображается через Phaser;
- двери и знаки над дверями отображаются через Phaser;
- дух появляется в Phaser;
- вопросы и результаты отображаются React-оверлеями;
- game domain остаётся источником истины.

## 19. Этап 13 — Effects MVP

Цель этапа: реализовать минимальную систему эффектов.

На этом этапе точные названия обычных бафов и дебафов всё ещё можно не фиксировать. Главное — механика.

Минимальные бафы:

- эффект, который убирает один неправильный вариант ответа;
- эффект, который даёт право на одну ошибку;
- эффект, который показывает информацию о знаке / двери;
- эффект, который снижает сложность следующей комнаты;
- эффект, который позволяет заменить вопрос.

Минимальные дебафы:

- эффект, который добавляет таймер;
- эффект, который скрывает информацию о знаке / двери;
- эффект, который повышает сложность следующей комнаты;
- эффект, который добавляет случайную технологию в пул;
- эффект, который отключает помощь.

Отдельный статус:

- Впечатление: слабое / ровное / сильное.

Минимальные HR-эффекты:

- Хороший контакт;
- Уверенная подача;
- Красный флаг;
- Неловкая пауза.

Необязательно для текущего MVP:

- Good Mood / Anxiety / Focus как отдельная mood-система;
- обязательное супер-проклятие;
- барабан наград;
- выбор награды из нескольких вариантов.

Результат этапа:

- эффекты реально меняют прохождение;
- игрок видит активные бафы и дебафы;
- Впечатление влияет на HR-комнату и финал;
- HR-комната имеет собственный пул эффектов.

---

## 20. Этап 14 — Final Interview

Цель этапа: сделать финальное собеседование настоящей целью run.

Задачи:

- создать финальную комнату;
- добавить финальный стол;
- добавить двух интервьюеров: HR и технического;
- выбрать серию вопросов подряд;
- использовать те же типы испытаний, что и в обычных комнатах;
- добавить более сложные и “душные” формулировки;
- учитывать накопленные эффекты;
- учитывать Впечатление;
- посчитать результат;
- показать passed / failed;
- показать финальную фразу;
- дать возможность выйти в меню или начать новый run.

Пример влияния Впечатления:

```txt
Сильное → 5 вопросов / можно 2 ошибки
Ровное → 6 вопросов / можно 1 ошибку
Слабое → 7 вопросов / можно 1 ошибку
```

Возможные финальные фразы:

- “Спасибо, мы свяжемся с вами до конца недели.”
- “Мы приняли решение продолжить с другими кандидатами.”

Результат этапа:

- run имеет финальную цель;
- игрок может выиграть или проиграть;
- финал связан с общей темой проекта;
- Впечатление ощущается как важный статус, а не декоративная подпись.

## 21. Этап 15 — Persistence Layer

Цель этапа: добавить минимальное локальное сохранение без backend и без привязки компонентов к конкретному хранилищу.

Для MVP используется localStorage.

Сохранять минимально:

- последний результат run;
- короткую историю последних run;
- глобальный язык интерфейса, отдельно от настроек run;
- последние настройки run.

Важно: localStorage не должен вызываться напрямую из случайных компонентов.

Нужно создать отдельный слой:

```txt
src/shared/storage/
```

Возможные функции:

```txt
saveLastRunResult()
getLastRunResult()
saveRunHistory()
getRunHistory()
saveRunSettings()
getRunSettings()
saveLanguage()
getLanguage()
```

Эти функции вызываются из repository/service слоя, а не напрямую из UI-компонентов.

Позже реализацию можно заменить:

- на IndexedDB, если локальных данных станет много;
- на backend, когда появятся аккаунты, история run и статистика;
- на гибридный вариант backend + локальный cache.

Результат этапа:

- после перезагрузки сохраняется минимум полезных данных;
- backend всё ещё не нужен;
- архитектура не мешает будущему сохранению на сервер;
- компоненты не зависят напрямую от localStorage.

---

## 22. Сериализуемый результат run

Результат run должен сразу проектироваться так, чтобы его можно было отправить на backend.

Минимальная структура результата:

```ts
type RunResult = {
  runId: string;
  seed: string;
  settings: RunSettings;
  startedAt: string;
  finishedAt: string;
  rooms: RoomResult[];
  effects: EffectSnapshot[];
  finalInterview: FinalInterviewResult;
  outcome: "passed" | "failed";
};
```

Важно хранить:

- seed run;
- выбранный пул технологий;
- последовательность комнат;
- id вопросов;
- ответы игрока;
- полученные эффекты;
- время начала и конца;
- итог run.

Это нужно, чтобы в будущем:

- сохранять историю run;
- показывать статистику;
- строить leaderboard;
- частично проверять честность результата;
- делать replay или подробный разбор run.

## 23. Будущая backend-интеграция

Или отдельный backend-пакет/сервис, если проект вырастет.

На backend уйдут:

- регистрация;
- авторизация;
- пользователи;
- история run;
- статистика;
- leaderboard;
- вопросы;
- модерация вопросов;
- пользовательские задачи;

Важно: backend не должен менять правила game domain. Он должен хранить данные, выдавать контент, принимать результаты и обслуживать продуктовые функции.

## 27. Что нельзя делать до готового core loop

До готового полного run нельзя делать:

- полноценный backend;
- авторизацию;
- регистрацию;
- профиль;
- leaderboard;
- админку;
- livecoding;
- sandbox для запуска кода;
- мета-прогрессию;
- магазин;
- достижения;
- большую карту;
- много языков программирования;
- разные финальные боссы;
- полноценные мини-боссы;
- мобильную версию.

Исключение: можно проектировать backend-ready интерфейсы и сериализуемые структуры данных, но нельзя уходить в полноценную backend-разработку до работающего core loop.
