# VISUAL_DIRECTION.md

> **Статус документа:** рабочий и изменяемый. Он фиксирует актуальное визуальное направление, но не является финальным набором неизменных условий. По ходу разработки визуальные решения, экраны и требования к ассетам могут добавляться, изменяться или удаляться по решению владельца проекта. После такого решения связанные документы должны быть синхронизированы с новой версией.

## 1. Назначение документа

Этот документ фиксирует визуальное направление **Code Dungeon** для MVP.

Цель документа — собрать в одном месте принятые решения по стилю, сцене, персонажам, UI, эффектам и основным игровым экранам, чтобы дальше не возвращаться к одним и тем же вопросам и использовать единое направление при разработке и генерации ассетов.

Документ не является финальным production art plan. Это визуальная база для первого playable prototype.

### Рабочие концепт-референсы

В репозитории сохранены примерные визуальные концепты. Это **не финальные макеты**, не точные UI-спецификации и не обязательный набор механик. Они задают общее настроение, композицию, визуальный язык и возможные состояния интерфейса.

Концепты могут содержать устаревшие, отложенные или ещё не подтверждённые элементы — например выбор сложности в Run Setup, дополнительные показатели HUD или разное количество дверей. Для игровых правил и состава MVP приоритет имеют актуальный gameplay-first план, MVP Scope и реальный код.

## 2. Общая визуальная идея

**Code Dungeon** — светлый sacred-tech roguelike про IT-собеседования.

Основной визуальный тон:

```txt
ivory / gold / white sacred-tech
чисто
светло
минималистично
мистически
не киберпанк
не fantasy dungeon
не реалистичный офис
```

Игра должна ощущаться как сакральный цифровой зал испытаний, где обычный кандидат проходит мифологизированное IT-собеседование.

Главный контраст:

```txt
мир — сакральный, чистый, мистический
герой — обычный человек дома в рубашке, шортах и тапочках
```

Этот контраст является частью идеи проекта, а не ошибкой стиля.

## 3. Камера и сцена

Игра использует **2D / 2.5D** вид с фиксированной камерой.

Фиксируем:

```txt
3/4 top-down камера
комната почти целиком видна
герой находится ниже центрального объекта
центральный объект в центре комнаты
двери сверху
без camera-follow за героем
```

Камера не должна превращать сцену в обычную изометрию с постоянным движением за персонажем. Для MVP сцена должна быть читаемой как единая комната.

## 4. Базовая комната

Базовая комната — светлая sacred-tech chamber.

Фиксируем:

```txt
белый / ivory / светло-серый камень
тонкие золотые линии
чистая геометрия
просторная центральная зона
архитектурные двери сверху
минимум визуального шума
```

Комната является базой для разных типов событий.

В обычной комнате в центре стоит алтарь / пьедестал.  
В HR-комнате вместо алтаря стоит стол HR.  
В финальной комнате вместо алтаря стоит финальный стол с двумя интервьюерами.

Сама базовая комната не должна каждый раз перерисовываться полностью. Меняется центральная фигура / объект и состояние сцены.

## 5. Герой

Герой — обычный remote-кандидат, попавший в сакральный dungeon-собеседование.

Фиксируем образ:

```txt
мужчина со спины
рубашка
шорты
тапочки
бытовой, уязвимый, немного нелепый
не герой фэнтези
не рыцарь
не маг
не sci-fi воин
```

Герой должен выглядеть человечески и немного иронично, но не как дешёвый мем.

Важная идея:

```txt
по ту сторону экрана сидит обычный человек,
а вокруг него система собеседований превращена в мистический roguelike-мир
```

## 6. Центральный объект обычной комнаты

Центральный объект обычной комнаты — алтарь / пьедестал.

Функции:

```txt
точка запуска испытания
точка появления технологии / символа
точка выдачи результата комнаты
точка выдачи бафа / дебафа / ничего
```

Поток обычной комнаты:

```txt
игрок входит
→ подходит к алтарю
→ алтарь активируется
→ появляется цифровое пламя / символ технологии
→ открывается challenge overlay
→ игрок проходит вопросы
→ overlay закрывается
→ алтарь выдаёт итог
→ баф / дебаф / ничего
→ появляются знаки над дверями
```

После испытания алтарь выдаёт только:

```txt
баф
дебаф
ничего
```

## 7. Двери и знаки выбора пути

Двери визуально не открываются.

После завершения комнаты появляются интерактивные знаки над доступными дверями.

Фиксируем:

```txt
до результата комнаты:
- знаков нет
- двери неактивны

после результата комнаты:
- над дверями появляются знаки
- знак может быть неизвестным или раскрытым
- игрок выбирает знак / дверь
- запускается переход
```

Основной интерактивный элемент выбора пути — **знак над дверью**, а не анимация открытия двери.

Door Insight раскрывает информацию именно на знаке.

Пример:

```txt
без Door Insight:
?   ?

с Door Insight:
JS  ?
HR  ?
```

На MVP знак показывает только тип / направление следующей комнаты, без перегруза сложностью, наградами и точными эффектами.

## 8. Духи технологий

Духи технологий существуют в первую очередь как overlay-портреты, а не как полноценные модели в комнате.

Фиксируем:

```txt
бело-золотые digital-mystic сущности
капюшоны / маски / пустые лица
сильный силуэт
без буквальных логотипов на теле
не реалистичные люди
не anime
```

В комнате дух может быть представлен через:

```txt
алтарь
цифровое пламя
символ технологии
портрет в overlay
```

JS Spirit принят как базовое направление для technology spirits.

## 9. Challenge Overlay

Challenge Overlay — React UI поверх Phaser-сцены.

Он открывается после активации центрального объекта.

Поддерживаемые типы испытаний:

```txt
Quiz
True / False
Code Output
Find the Bug
Choose Correct Fragment
Order Steps
```

Overlay должен поддерживать:

```txt
длинный вопрос
code block
4–6 вариантов ответа
длинные варианты ответа
номер вопроса внутри комнаты
технологию
тип вопроса
сложность
портрет духа
```

Главное правило:

```txt
после отдельного ответа не показываем правильность
не показываем correct answer
не показываем explanation
```

Ответ записывается, затем открывается следующий вопрос.

Итог показывается только после завершения всей комнаты.

## 10. Итог комнаты

Результат считается после всей серии вопросов в комнате.

Возможные исходы:

```txt
обычное прохождение → ничего
провал → дебаф
идеальное прохождение → баф
```

После результата:

```txt
алтарь активируется
→ появляется цифровое пламя
→ если есть эффект, появляется иконка
→ иконка улетает в HUD
→ появляются знаки над дверями
```

Награда является финальной фазой текущей комнаты, а не отдельным выбором игрока.

Эффекты делятся на:

```txt
бафы
дебафы
HR-бафы
HR-дебафы
```

Бафы и дебафы отображаются в HUD отдельными строками.

Фиксируем стиль иконок:

```txt
круглая sacred-tech иконка
тёмная читаемая основа
бафы — зелёно-золотой / позитивный акцент
дебафы — красный / фиолетово-красный тревожный акцент
символ должен читаться в маленьком размере
```

Иконки должны быть понятны как положительные / отрицательные даже без чтения описания.

## 12. HUD во время run

HUD состоит из постоянной компактной кнопки с иконкой головы и раскрываемой панели поверх Phaser-сцены.

Базовое поведение:

```txt
иконка головы видна постоянно
вне испытания игрок может открыть или закрыть HUD по нажатию на иконку
во время активного испытания HUD принудительно открыт
во время активного испытания игрок не может закрыть HUD
после завершения испытания ручное переключение снова доступно
```

Механически HUD показывает:

```txt
жизнь 1/1
бафы
дебафы
Впечатление
пауза / меню
```

Бафы и дебафы отображаются иконками в отдельных строках.  
Подробности открываются через tooltip.

Сам HUD не показывает данные текущего вопроса. Во время испытания эти данные отображаются отдельным Challenge Overlay, а HUD остаётся открытым рядом с ним.

Не показываем в HUD:

```txt
тип вопроса
технологию вопроса
сложность вопроса
номер вопроса внутри комнаты
таймер
подсказки
варианты ответа
```

Эти данные относятся к Challenge Overlay.

Визуальное решение HUD пока не финализировано. Ориентируемся на примерные концепты панели эффектов и Challenge Overlay, но их композиция и набор показателей не являются точной спецификацией.

## 13. HR-комната

HR-комната использует ту же базовую комнату.

Отличие:

```txt
в центре стоит рабочий стол HR
за столом сидит HR-сущность
игрок подходит к столу снизу
открывается HR overlay
```

В HR-комнате нет кодового испытания и нет выбора ответа.

HR проверяет soft-skill статус игрока — **Впечатление** — и выдаёт результат.

## 14. Впечатление

Впечатление — soft-skill статус игрока.

Это не настроение и не обычный баф/дебаф.

Фиксируем уровни:

```txt
-1 → Слабое впечатление
 0 → Ровное впечатление
+1 → Сильное впечатление
```

Как меняется:

```txt
провал комнаты → Впечатление ухудшается
обычное прохождение → Впечатление не меняется
идеальное прохождение → Впечатление улучшается
```

Как влияет на HR:

```txt
Слабое впечатление → HR-дебаф
Ровное впечатление → ничего
Сильное впечатление → HR-баф
```

Как влияет на финал:

```txt
Сильное впечатление → финал мягче
Ровное впечатление → стандартный финал
Слабое впечатление → финал жёстче
```

## 15. HR-эффекты

HR-эффекты — отдельные эффекты, которые можно получить только в HR-комнате.

В HUD они отображаются как обычные бафы / дебафы, но имеют источник `hr`.

MVP-набор:

```txt
HR-бафы:
- Хороший контакт
- Уверенная подача

HR-дебафы:
- Красный флаг
- Неловкая пауза
```

Черновая логика:

```txt
Хороший контакт → блокирует следующий HR-дебаф
Уверенная подача → следующий провал комнаты не ухудшает Впечатление
Красный флаг → следующее идеальное прохождение не улучшает Впечатление
Неловкая пауза → следующий провал сразу ставит Слабое впечатление
```

## 16. Финальное собеседование

Финальная комната — отдельная версия базовой комнаты.

В центре стоит финальный стол.  
За столом сидят два интервьюера:

```txt
HR-интервьюер
технический интервьюер
```

Игрок подходит снизу, после чего открывается final interview overlay.

Финал использует те же типы вопросов, что и обычные комнаты, но вопросы сложнее.

Финальная сложность строится из двух частей:

```txt
техническая сложность
+
интервьюерская душность
```

Интервьюерская душность — это вопросы с похожими вариантами ответа, где правильность зависит от точной формулировки.

Пример логики:

```txt
“исключительно”
“только в этом случае”
“наиболее точный вариант”
“какой вариант нельзя считать правильным”
```

После финала нет бафов, дебафов, дверей и перехода дальше.

Поток:

```txt
финальное испытание
→ подсчёт результата
→ экран результата run
→ меню / новый run
```

## 17. Экран результата run

Экран результата принят как визуальное направление.

Фиксируем:

```txt
светлый ivory/gold sacred-tech UI
крупный статус Passed / Failed
финальная фраза
портреты HR и технического интервьюера в верхней зоне
статистика run
блок Впечатления
полученные бафы / дебафы
технологии в run
кнопки “В меню” и “Новый run”
```

Даже победа не должна звучать как обычное “YOU WIN”.

Финальная фраза при успехе:

```txt
“Спасибо, мы свяжемся с вами до конца недели.”
```

При провале:

```txt
“Мы приняли решение продолжить с другими кандидатами.”
```

## 18. Run Setup

Run Setup — экран подготовки кандидата перед забегом.

Визуально берём за основу стиль финального экрана:

```txt
два интервьюера сверху
светлый ivory/gold UI
крупный заголовок
чистые панели настроек
большая кнопка “Начать run”
```

Игрок выбирает:

```txt
формат пула: Frontend / Custom

```

Язык интерфейса задаётся глобально и не дублируется на этом экране. Выбора базовой сложности в Run Setup пока нет.

После нажатия “Начать run” игрок попадает не в pop-up, а в первую стартовую комнату.

## 19. Первая комната / стартовый баф

Run начинается с первой стартовой комнаты.

Поток:

```txt
Run Setup
→ Начать run
→ первая комната
→ игрок подходит к алтарю
→ получает случайный стартовый баф
→ баф добавляется в HUD
→ появляются знаки над дверями
→ игрок выбирает следующую комнату
```

Первая комната не содержит испытания.

Её задача:

```txt
выдать стартовый баф
показать HUD
дать первый выбор пути
```

Стартовый баф не выбирается игроком вручную.

## 21. Переходы

Все переходы в MVP строятся на простых анимациях:

```txt
fade
glow
pulse
screen flash
icon fly-to-HUD
```

Между web-экранами:

```txt
fade out
→ смена экрана
→ fade in
```

Между комнатами:

```txt
выбор знака над дверью
→ знак вспыхивает
→ fade out
→ новая комната
→ fade in
```

Запуск испытания:

```txt
подход к центральному объекту
→ активация
→ цифровое пламя / символ
→ challenge overlay
```

После испытания:

```txt
overlay закрывается
→ алтарь выдаёт баф / дебаф / ничего
→ появляются знаки дверей
```

Не делаем:

```txt
длинные катсцены
3D-переходы
анимацию открытия дверей
отдельный loading screen между каждой комнатой
```

## 23. Правила ассетов

Общие правила:

```txt
PNG — для персонажей, комнаты, столов, духов, сложных painted-ассетов
SVG — для простых знаков, UI-иконок, технологических символов
прозрачный фон — для объектов, персонажей, духов, иконок
без текста внутри игровых ассетов, если текст можно сделать в UI
без водяных знаков
без случайных логотипов
```

UI-текст должен быть настоящим текстом в React, а не частью картинки.

Исключения возможны только для логотипа.

## 25. Главный принцип дальше

Не расширять визуальный scope, пока не собран playable loop.

Приоритет:

```txt
игровая сцена
run setup
первая комната со стартовым бафом
обычная комната с испытанием
итог комнаты
знаки дверей
HR-комната
финальное собеседование
экран результата
```

Визуал должен помогать собрать playable prototype, а не уводить проект в бесконечную генерацию ассетов.
