# Project Concept

> **Статус документа:** рабочий и изменяемый. Он описывает актуальную концепцию, но не является финальным набором неизменных условий. По ходу разработки идеи, механики и границы проекта могут добавляться, изменяться или удаляться по решению владельца проекта. После такого решения связанные документы должны быть синхронизированы с новой версией.

## 1. Краткое описание

Проект — браузерная roguelike-игра про программирование.

Игрок проходит процедурно сгенерированный забег по комнатам. Вместо классических монстров в комнатах появляются духи языков программирования, технологий и IT-сущностей. Они дают игроку испытания: вопросы, квизы, задачи на понимание кода, поиск ошибок, выбор правильного фрагмента или расстановку шагов.

Проект в первую очередь является игрой, а не линейным обучающим курсом. Образовательная ценность появляется через игровой челлендж, объяснения после ответов и повторяемость забегов.

Главная цель — сделать реиграбельный programming roguelike, где игроку хочется снова начать забег, собрать другой билд, рискнуть с более сложными комнатами и пройти дальше.

---

## 2. Главная идея

Игрок начинает run с выбранными стартовыми настройками:

- способ формирования пула вопросов;
- конкретный набор технологий.

Язык интерфейса является глобальной настройкой приложения и не входит в параметры run. Выбор базовой сложности перед стартом пока не используется.

Пул вопросов можно сформировать двумя способами:

1. Выбрать готовое направление, например Frontend, Backend, Fullstack, DevOps или Algorithms. В этом случае игра сама собирает стартовый набор технологий для run.
2. Собрать кастомный пул вручную, выбрав конкретные языки, фреймворки и темы.

Дальше игра строится на случайности и выборе:

- случайная генерация комнат;
- случайные духи языков и технологий;
- случайные типы испытаний;
- случайная сложность вопросов;
- случайные награды, бафы, дебафы и супер-проклятия;
- выбор знаков над дверями после прохождения комнаты;
- неизвестность того, что находится за следующей дверью.

Игрок собирает билд во время забега. Бафы, дебафы, супер-проклятия и события меняют правила прохождения: могут убирать варианты ответа, добавлять право на ошибку, включать таймер, подмешивать незнакомые языки, менять язык интерфейса/ответов, показывать информацию о дверях или, наоборот, усложнять выбор.

Ключевая идея: игрок не просто отвечает на вопросы, а выживает внутри непредсказуемого собеседовательного dungeon-run.

## 3. Приоритет проекта

Главный приоритет — геймплей.

Проект не должен ощущаться как обычный тест, викторина или обучающий тренажёр. Вопросы и задачи являются частью боевой системы, но не заменяют игру целиком.

Правильное ощущение:

> “Я прохожу опасный забег, собираю билд, рискую, принимаю решения и сталкиваюсь с непредсказуемыми испытаниями.”

Неправильное ощущение:

> “Я просто отвечаю на вопросы по программированию на красивом фоне.”

## 4. Жанр и референсы

Жанр:

- browser game;
- roguelike / roguelite;
- programming challenge;
- quiz-based combat;
- procedural run;
- build-based progression.

Геймдизайнерские референсы:

- Slay the Spire — структура забега, выбор пути, билды, риск/награда;
- roguelike-игры — случайность, повторяемость, сложность, неожиданные события;
- programming interview / coding challenge — тематическая основа испытаний.

Визуальные референсы:

- Transistor — камера, ощущение комнаты, top-down / isometric-подача;
- 33 Immortals — ощущение персонажа, стилизация, читаемый вид сверху;
- visual novel overlays — диалоги с духами и события поверх игровой сцены.

## 5. Целевая аудитория

Проект рассчитан на программистов и людей, связанных с разработкой.

В долгосрочной перспективе игра может поддерживать разные направления:

- frontend;
- backend;
- fullstack;
- algorithms;
- DevOps;
- database;
- language-specific runs.

Для MVP стартовый фокус ограничивается frontend/fullstack-базой, чтобы не распылять разработку и контент.

Стартовый пул тем:

- JavaScript;
- TypeScript;
- HTML;
- CSS;
- React;
- Vue;
- Git.

Позже можно добавить:

- Node.js;
- SQL;
- Python;
- PHP;
- Java;
- C#;
- Go;
- Docker;
- Linux;
- Algorithms;
- System Design.

## 6. Формирование пула вопросов

Перед началом run игрок выбирает, как будет сформирован пул вопросов.

### Готовое направление

Игрок выбирает направление, а игра сама собирает набор технологий.

Пример направления Frontend:

- JavaScript;
- TypeScript;
- HTML;
- CSS;
- React;
- Vue;
- Git;
- Browser API.

Это быстрый вариант старта для игрока, который не хочет вручную выбирать каждую тему.

### Кастомный пул

Игрок сам выбирает конкретные технологии и темы.

Например:

- JavaScript;
- TypeScript;
- React;
- CSS;
- Git.

Это вариант для игрока, который хочет сам контролировать, что может выпадать в run.

Важно: выбранный пул не является абсолютно безопасным. Во время забега дебафы, HR-события, супер-проклятия и особые комнаты могут временно или постоянно добавить в пул незнакомую технологию.

Пример:

Игрок выбрал Frontend, но получил дебаф, который добавляет SQL или PHP-вопросы. Это не ошибка баланса, а часть roguelike-челленджа.

## 7. Core Gameplay Loop

Базовый цикл забега:

1. Игрок начинает новый run.
2. Выбирает способ формирования пула вопросов.
3. Выбирает готовое направление или собирает кастомный пул.
4. Попадает в первую стартовую комнату.
5. Подходит к алтарю.
6. Получает случайный стартовый баф.
7. После получения бафа над дверями появляются знаки.
8. Игрок выбирает знак / дверь.
9. За дверью находится следующая случайная комната.
10. В обычной комнате игрок активирует центральный объект.
11. Открывается challenge overlay.
12. Игрок отвечает на вопросы комнаты.
13. После каждого отдельного ответа правильность не показывается сразу.
14. После завершения всей комнаты считается общий результат.
15. Алтарь выдаёт баф, дебаф или ничего.
16. Впечатление игрока может измениться.
17. Над дверями появляются новые знаки выбора пути.
18. Цикл повторяется до HR-событий, финального собеседования, поражения или завершения run.

---

## 8. Финальная цель run

Финальный босс забега — собеседование на работу.

Это не обычный монстр, а финальное испытание, где игрок должен пройти собеседование после всех комнат, бафов, дебафов, HR-событий и накопленного Впечатления.

Базовый финал:

- игрок доходит до финальной комнаты;
- в центре находится финальный стол;
- за столом сидят два интервьюера: HR и технический интервьюер;
- начинается собеседование;
- собеседование включает несколько вопросов, технологий или типов испытаний;
- накопленные бафы, дебафы и Впечатление влияют на прохождение;
- даже после успешного прохождения игра может завершаться фразой в духе: “Спасибо, мы свяжемся с вами до конца недели”;
- после этого игрок может начать следующий run.

В будущем финальные боссы и мини-боссы могут быть разными:

- обычный интервьюер;
- два душных senior-разработчика;
- тимлид;
- CTO;
- владелец компании;
- HR-финалист;
- техническая комиссия;
- legacy-архитектор;
- автор тестового задания на 40 часов.

Идея финала должна сохранять иронию IT-собеседований, но при этом оставаться частью серьёзной roguelike-системы.

## 9. Типы комнат

### Боевые комнаты

Основной тип комнаты. Игрок встречает духа языка или технологии и проходит испытание.

Примеры:

- JavaScript Spirit;
- TypeScript Spirit;
- CSS Spirit;
- React Spirit;
- Vue Spirit;
- Git Spirit.

### Комнаты наград

Комнаты наград остаются идеей для будущего расширения.

В текущем MVP отдельной Reward Room нет: бафы и дебафы выдаются через алтарь после результата комнаты, а стартовый баф выдаётся в первой комнате.

В будущем игрок может выбирать один из нескольких бафов, артефактов или положительных модификаторов.

### Комнаты негативных событий

Игрок получает дебаф, сложное условие или неприятное изменение run.

Важно: обычные негативные события не должны выдавать любые случайные эффекты из общего мешка. У каждого типа комнаты должен быть свой пул эффектов.

### HR-комнаты

Случайные события с HR-персонажами. HR-комнаты имеют свой отдельный пул эффектов.

HR может:

- навесить специфический HR-дебаф;
- изменить условия следующего собеседования;
- добавить странное ограничение;
- скрыть часть информации;
- изменить язык вариантов ответа;
- выдать неожиданный бонус при наличии подходящего состояния игрока.

Пример:

В текущем MVP HR смотрит на статус “Впечатление”: слабое даёт HR-дебаф, ровное даёт нейтральный исход, сильное даёт HR-баф.

### Комнаты настроения

Комнаты настроения остаются идеей для future expansion.

В текущем MVP отдельной Mood Room нет: её роль заменяет статус “Впечатление”, который меняется после результатов обычных комнат и влияет на HR / финал.

В будущем это могут быть события, которые временно меняют состояние игрока.

Примеры:

- хорошее настроение;
- уверенность;
- выгорание;
- тревожность;
- синдром самозванца;
- концентрация.

Такие состояния могут влиять на будущие комнаты, HR-события, награды или финальное собеседование. Для текущего MVP вместо этой системы используется Впечатление.

### Комнаты старого знакомого / бывшего друга

Особые случайные встречи. Например, игрок встречает бывшего друга, коллегу, наставника или странного знакомого из прошлого.

Такая комната может:

- усилить игрока;
- дать редкий баф;
- убрать дебаф;
- открыть особую дверь;
- изменить будущий пул событий;
- дать неожиданный риск.

### Секретные комнаты

Редкие комнаты с нестандартными наградами, событиями или рисками.

### Мини-босс комнаты

Особые сложные комнаты перед финалом.

Возможные мини-боссы:

- душный senior;
- reviewer, который требует идеальный кодстайл;
- legacy maintainer;
- HR screening;
- team lead;
- live coding ghost, даже если livecoding ещё не является основной механикой.

### Финальная комната

Финальное собеседование на работу.

## 10. Типы испытаний

Для MVP используются задания без livecoding.

Стартовые типы:

### Quiz

Классический вопрос с вариантами ответа.

### True / False

Утверждение, которое нужно подтвердить или опровергнуть.

### Code Output

Игроку показывается фрагмент кода. Нужно выбрать, что он выведет.

### Find the Bug

Игроку показывается код или описание ситуации. Нужно найти ошибку.

### Choose Correct Fragment

Игрок выбирает правильный вариант кода или решения.

### Order Steps

Игрок расставляет шаги в правильном порядке.

Livecoding не входит в MVP. Его можно добавить позже как отдельный сложный режим с запуском кода и тестами.

---

## 11. Сложность

Сложность является частью геймплея, а не стартовой настройкой run.

Игрок пока не выбирает Easy / Normal / Hard перед началом забега. Сложность конкретных вопросов и комнат определяется game logic и может меняться из-за:

- дебафов;
- супер-проклятий;
- комнаты;
- выбранного пути;
- случайного события;
- типа духа;
- текущего билда;
- выбранного пула технологий.

Сложность должна работать не только как число, но и как набор условий.

Примеры усложнения:

- больше вопросов в комнате;
- меньше времени на ответ;
- выше сложность темы;
- меньше вариантов помощи;
- незнакомая технология добавляется в пул;
- сложные типы заданий выпадают чаще;
- ошибка сильнее влияет на run.

## 12. Система эффектов

В игре должны быть разные типы эффектов. Их нельзя смешивать в один общий список.

Каждый эффект должен иметь:

- тип;
- источник;
- условия выпадения;
- силу;
- длительность;
- область влияния;
- редкость;
- совместимость с другими эффектами.

Базовые категории эффектов:

### Бафы

Положительные эффекты.

Игрок может получить баф:

- в начале run;
- за идеально пройденную комнату;
- за успешное событие;
- от редкого персонажа;
- через особую синергию.

Примеры бафов:

- убрать один неправильный вариант ответа;
- дать право на одну ошибку;
- показать сложность комнаты перед входом;
- показать тип комнаты за дверью;
- заменить сложный вопрос на другой;
- заблокировать один нежелательный язык на несколько комнат;
- получить дополнительную награду за серию правильных ответов;
- усилить финальное собеседование;
- ослабить следующий HR-дебаф.

### Дебафы

Обычные негативные эффекты.

Игрок может получить дебаф:

- за провал комнаты;
- за плохое событие;
- от HR-комнаты;
- от рискованного выбора;
- от некоторых дверей;
- от мини-босса.

Примеры дебафов:

- добавить таймер на комнаты;
- добавить случайный язык в пул;
- скрыть варианты ответа до выбора;
- перемешать описание вопроса;
- увеличить минимальную сложность вопросов;
- уменьшить количество доступных ошибок;
- добавить обязательную комнату с незнакомой технологией;
- скрыть информацию о дверях;
- усложнить следующий вопрос.

### Супер-проклятия

Сильные негативные эффекты, которые игрок получает не за обычную ошибку, а за серьёзный провал.

Супер-проклятие может выдаваться:

- за потерю жизни;
- за критический провал комнаты;
- за провал важного испытания;
- после мини-босса;
- как цена за очень сильную награду.

Супер-проклятия должны сильно менять run и создавать ощущение последнего шанса.

Примеры супер-проклятий:

- все следующие комнаты получают повышенную сложность;
- в пул добавляется незнакомая технология до конца run;
- игрок теряет часть накопленных бафов;
- финальное собеседование становится сложнее;
- часть вариантов ответа скрыта;
- каждый следующий провал усиливает текущее проклятие;
- двери перестают показывать любую информацию.

### Состояния

Временные или условные эффекты, которые сами по себе не являются бафом или дебафом, но влияют на будущие события.

Примеры состояний:

- хорошее настроение;
- уверенность;
- концентрация;
- выгорание;
- тревожность;
- синдром самозванца.

Пример синергии:

В текущем MVP такой условный статус упрощён до Впечатления: сильное Впечатление помогает в HR-комнате и финале, слабое Впечатление усложняет HR и финал.

## 13. Источники эффектов

У каждого эффекта должен быть конкретный источник.

Это важно для баланса, архитектуры и ощущения честности.

Примеры источников:

### Start Pool

Эффекты, которые могут выпасть в начале run.

Например:

- стартовый баф;
- стартовый артефакт;
- стартовый модификатор.

### Perfect Room Pool

Эффекты за идеальное прохождение комнаты.

Например:

- бонусный баф;
- временное усиление;
- дополнительный выбор награды.

### Failed Room Pool

Эффекты за провал комнаты.

Например:

- обычный дебаф;
- потеря ресурса;
- ухудшение следующей комнаты.

### Life Loss Pool

Эффекты за потерю жизни или критический провал.

Например:

- супер-проклятие;
- тяжёлый постоянный модификатор;
- усиление финального собеседования против игрока.

### HR Pool

Эффекты, которые может выдать только HR-комната.

Например:

- HR-дебаф;
- изменение условий финала;
- странное ограничение;
- редкий HR-супер-баф при особых условиях.

### Impression Pool / Mood Pool

Для текущего MVP используется Впечатление: слабое, ровное, сильное.

Mood Pool с хорошим настроением, концентрацией, тревожностью и выгоранием можно добавить позже как расширение системы состояний.

### Special Encounter Pool

Эффекты от особых встреч.

Например:

- бывший друг;
- старый коллега;
- наставник;
- странный рекрутер;
- случайный open-source maintainer.

Такой подход нужен, чтобы эффекты не выпадали хаотично. Игрок должен чувствовать, что мир случайный, но не бессмысленный.

---

## 14. Жизни и провал

Систему жизней нужно отдельно доработать.

Предварительная идея:

- у игрока есть ограниченный ресурс выживания;
- обычная ошибка может дать дебаф или ухудшить состояние комнаты;
- провал комнаты может снять жизнь или дать сильный дебаф;
- потеря жизни не обязательно завершает run сразу;
- при потере жизни игрок получает супер-проклятие и продолжает run на последнем шансе;
- полный проигрыш наступает, когда игрок больше не может продолжать run.

Важно: поражение не должно быть только “экраном смерти”. Даже ошибка должна создавать новую драму и менять билд.

---

## 15. Языки интерфейса и локализация

Проект сразу закладывается под два основных языка:

- русский;
- английский.

Язык интерфейса выбирается глобально для приложения. Он не является параметром отдельного run и не дублируется в Run Setup.

Локализация должна поддерживать:

- интерфейс;
- описания бафов/дебафов;
- диалоги;
- вопросы;
- варианты ответа;
- объяснения;
- названия комнат;
- названия духов.

Нужно разделять:

- interface language — язык интерфейса;
- content language — язык задач;
- modifier language — временный язык, который может быть изменён дебафом.

Пример языкового дебафа:

- изменить язык вариантов ответа;
- заменить часть интерфейса или ответов на другой язык;
- включить “lost in translation” эффект.

## 16. Визуальная концепция

Игра использует 2D/2.5D вид сверху или изометрическую камеру.

Настоящее 3D на старте не используется.

Игровая сцена:

- комната;
- персонаж;
- двери;
- центральный объект: шар, постамент, терминал или место появления духа;
- дух языка/технологии;
- эффекты появления;
- переходы между комнатами.

Диалоги с духами и событиями показываются через overlay в стиле visual novel: поверх игровой сцены появляется портрет/силуэт духа, текст, варианты ответа или начало испытания.

Проект может иметь светлую и тёмную тему. Визуальный стиль должен быть достаточно простым для реализации, но не выглядеть как сырой прототип.

## 24. Главные риски

### Контент

Нужно много качественных вопросов и задач. Без хорошего контента игра быстро станет скучной.

### Баланс

Бафы, дебафы, супер-проклятия, состояния, сложность и рандом должны создавать интересный челлендж, а не ощущение несправедливости.

### Скоуп

Проект легко раздуть. Нужно строго отделять MVP от будущих идей.

### Архитектура

Игровая логика не должна быть размазана по React-компонентам. Нужен отдельный game layer.

### Система эффектов

Если не разделить источники эффектов, игра быстро превратится в хаос. Нужно сразу проектировать пулы эффектов: стартовые, комнатные, HR, провальные, супер-проклятия, особые встречи.

### Мотивация

Проект большой. Нужно идти маленькими рабочими итерациями: сначала playable prototype, потом контент, потом backend, потом продуктовые фичи.

## 25. Текущее зафиксированное решение

На текущем этапе зафиксировано:

- проект является игрой в первую очередь;
- жанр — browser programming roguelike;
- обучение вторично;
- gameplay loop важнее линейного обучения;
- стартовый стек — Next.js + TypeScript + Phaser;
- livecoding не входит в MVP;
- первый набор испытаний — quiz, true/false, code output, find bug, choose fragment, order steps;
- визуал — светлый ivory/gold sacred-tech 2D/2.5D top-down;
- стартовый фокус — frontend/fullstack темы;
- перед run есть два способа формирования пула вопросов: готовое направление или кастомный пул;
- отдельных режимов прохождения пока нет;
- языки интерфейса — русский и английский;
- run начинается со стартовой комнаты;
- стартовый баф выдаётся через алтарь в первой комнате;
- Reward Room не входит в текущий MVP как отдельная комната;
- Mood Room не входит в текущий MVP как отдельная комната;
- после отдельного ответа не показываем correct/wrong и explanation;
- итог считается после всей комнаты;
- обычная комната выдаёт баф / дебаф / ничего через алтарь;
- основной выбор пути — знаки над дверями;
- двери визуально не открываются;
- статус Впечатление заменяет mood-систему в текущем MVP;
- HR-комнаты имеют собственный пул эффектов и смотрят на Впечатление;
- финальный босс — собеседование на работу;
- финал включает двух интервьюеров: HR и технического;
- Впечатление влияет на финальное собеседование;
- эффекты делятся на обычные бафы, обычные дебафы, HR-бафы и HR-дебафы;
- супер-проклятие / Last Chance остаётся optional или future expansion;
- у каждого эффекта должен быть конкретный источник выпадения;
- backend нужен, но его можно подключать постепенно;
- контент задач — главный риск и отдельная важная система;
- актуальное визуальное направление собрано в `VISUAL_DIRECTION.md`.
